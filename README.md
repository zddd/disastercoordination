# 灾害应急调度中心 (Disaster Coordination Center)

> 连接受灾群众与救援力量 — 从求助上报到救援完成的全链路平台

---

## 项目背景

一直以来，我们几乎每年都发生多起不同类型的重大灾害：

- **瘟疫**：SARS、新冠疫情……
- **地震**：长宁 6.0 级、泸定 6.8 级、积石山 6.2 级、日喀则 6.8 级……
- **特大洪涝**：长江淮河流域、郑州 7·20、京津冀洪涝、广西洪灾……
- **超强台风**：利奇马、摩羯……
- **重大事故**：响水爆炸、长沙自建房倒塌、天津港爆炸……

2024 年全国自然灾害造成 **9413 万人次受灾**，因灾死亡失踪 **856 人**，直接经济损失超 **3000 亿元**。

### 每次灾害后，救助环节都面临同样的结构性困境

| 痛点 | 表现 |
|------|------|
| **求助信息高度分散** | 灾后大量求救信息散落在微信/微博/QQ/抖音等社交渠道，无法统一收拢、无法有效响应 |
| **协作工具效率低下** | 在线文档作为"救命文档"虽是创举，但全员编辑容易混乱、移动端体验差、富媒体支持弱，不适合做任务调度 |
| **指挥缺乏全局视角** | 分散的信息难以纳入统筹，指挥方对灾情态势的掌握是模糊、片面的 |
| **每次重头开始** | 每次新灾情，熟悉的组织动员又从头来一遍，难以复用过往经验 |
| **触达覆盖率不足** | 预警和求助入口无法高效触达真正需要帮助的群体 |

### 核心机会

1. **QQ + 微信覆盖绝大多数中国人**，是独特的高触达基础设施
2. 灾害有其**模式化特征**——地震引发滑坡/危化品泄漏、洪涝引发溃坝/疫情——可构建为预案知识
3. 多次灾害中积累了丰富的实操经验，具备**系统化沉淀**的条件

---

## 项目目标

构建一套 **高覆盖、可复用、全链路** 的灾害应急调度系统，解决三大核心问题：

```
          需要救助的人
               ↕
  救援力量（救援队/志愿者）
               ↕
           物资资源
```

**核心价值主张**：

- 🚨 **让求助不再石沉大海**：统一求助入口，GPS 自动定位，一键求救
- 📊 **让指挥不再盲人摸象**：实时看板、热力地图、调度池、超时告警
- 🔍 **让信息不再真假难辨**：AI 预审 + 人工审核 + 信用分体系 + 孤立上报标记
- 🔄 **让经验不再从零开始**：灾害可归档、预案可复用、历史可对比

| **救援队** | 接收调度指令 → 导航到救援点 → 更新任务状态 → 团队内部协调 | H5 | ✅ |

| **指挥员** | 全局态势感知 → 资源调度决策 → 预案管理 → 灾情结束判定 | Web 管理后台 | ✅ |

| 角色 | 核心需求 | 使用端 | MVP 状态 |

| **值班员** | 灾害实例审核开通/关闭 → 预警响应确认 → 交接班记录 | Web 管理后台 | ✅ 部分 |

### 第三层：外围参与角色

| 角色 | 核心需求 | 使用端 | MVP 状态 |
|------|---------|--------|---------|
| **捐赠者** | 在线捐款/捐物 → 查看资金物资流向 → 接收反馈 | H5 | 📋 完整版 |
| **系统管理员** | 用户权限管理 → 系统配置 → 字典维护 → 数据归档 | Web 管理后台 | ✅ |

### 关键设计决策

- **分区指挥**：大范围灾害分行政区独立调度，总指挥可跨区协调
- **值班员**：7×24 小时灾害应急，换班信息交接防"掉棒"
- **信息审核员**：灾害期间信息噪音极高，进入调度池的求助必须经过验证
- **民间救援力量**：注册救援队 + 民间救援队两种类型，认证标识区分，调度能力相同

## 核心功能

### 完整业务链路（六阶段闭环）

```
预警阶段 → 求助上报阶段 → 信息核实阶段 → 救援调度阶段 → 物资保障阶段 → 闭环归档阶段
```

### MVP 功能矩阵 (v0.1.0)

| 模块 | 功能 | 用户 |
|------|------|------|
| **求助上报** | 按灾害类型动态切换表单（地震/洪涝/台风）、GPS 双源定位、一键求救 SOS、现场照片上传、弱网离线暂存 | 受灾群众 |
| **状态追踪** | 9 步进度条（已提交→已核实→救援中→已获救）、通过求助编号实时查询、SSE 实时推送状态变更 | 受灾群众 |
| **AI 预审** | 4 条确定性规则引擎（文本相似度 >90%、GPS 异常、同设备刷量、敏感词标记）| 系统 |
| **信息审核** | 审核队列按紧急度排序、SLA 监控（5/30/120 分钟）、通过/驳回/合并操作、驳回降信用分 | 审核员 |
| **指挥看板** | 核心指标卡片（求助总数/已响应/完成率）、调度池列表、按灾害切换、超时告警 | 指挥员 |
| **任务调度** | 手工分派救援队、批量分派（同区域同类型）、基于 PostGIS 距离推荐附近救援队 | 指挥员 |
| **任务状态机** | 11 种状态（分配→接单→前往→到达→施救→完成/无法完成/需增援），状态历史 JSONB 全量记录 | 救援队 |
| **救援队管理** | 注册审核流程、能力标签（水域/山地/医疗等）、类型标识（注册/民间）、GPS 位置上报 | 管理员 |
| **灾害管理** | 创建/关闭灾害、灾情总结报告（求助数/获救人数/队伍投入）、30 天自动归档 | 管理员 |
| **文件安全** | 6 层防护：类型白名单/Magic Bytes 校验/大小限制/路径穿越防护/UUID 重命名/IP 频率限制 | 系统 |
| **认证鉴权** | JWT (HS256, 24h)、bcrypt 密码哈希、10 角色 RBAC、三级权限分层（公开/认证/管理员） | 系统 |

## 快速开始

### 环境要求

| 工具 | 最低版本 | 安装方式 |
|------|---------|---------|
| Go | 1.22+ | `brew install go` 或 [go.dev/dl](https://go.dev/dl) |
| Node.js | 20+ | `nvm install 20` |
| Docker Desktop | 任意 | [docker.com](https://www.docker.com/products/docker-desktop/) |

### 一键启动（推荐）

```bash
cd /path/to/project

# 开发模式（自动建表 + 插入演示账号）
./scripts/start.sh

# 访问 http://localhost:3000/login
```

启动脚本自动完成：环境检查 → 启动 PostgreSQL/MinIO → 编译后端 → 自动建表 + 种子数据 → 启动前端。

其他命令：
```bash
./scripts/start.sh --reset           # 清空数据库重来
./scripts/start.sh --prod            # 生产模式（无演示账号）
./scripts/start.sh stop              # 停止所有服务
./scripts/start.sh restart           # 同时重启前后端
./scripts/start.sh restart-backend   # 单独重启后端
./scripts/start.sh restart-frontend  # 单独重启前端
```

## 使用流程

### 完整救援流程（8 步）

```
1. 管理员登录 → 创建灾害实例
2. 受灾群众打开 /help/submit → 选择灾害 → 提交求助 / 一键求救
3. 系统 AI 自动预审（异常坐标 / 重复 / 敏感词标记）
4. 审核员登录 → 审核通过 / 驳回求助
5. 审核通过 → 求助进入调度池
6. 指挥员登录 → 看板查看调度池 → 指派救援队
7. 救援队登录 → 接单 → 出发 → 到达 → 施救 → 完成
8. 受灾群众通过求助编号实时追踪进度
```

### 各角色操作指南

<details>
<summary><b>受灾群众 — 发起求助</b></summary>

1. 打开 `http://localhost:3000/help/submit`
2. 从下拉框选择当前活跃灾害
3. 选择求助类型（被困/受伤/缺水等，按灾害类型自动切换）
4. 选择紧急程度（紧急/一般/轻微）
5. （可选）填写详细描述、上传现场照片
6. 点击"提交求助"
7. 记录返回的求助编号 → 在 `/help` 页面输入编号追踪进度

> 💡 紧急情况直接点击"一键求救 SOS"，无需填写任何信息！
</details>

<details>
<summary><b>指挥员 — 调度救援</b></summary>

1. 登录 → 自动进入指挥看板
2. 顶部下拉框选择活跃灾害
3. 查看核心指标（求助总数/紧急待处理/一般/轻微）
4. 调度池区域查看待分配的求助
5. 点击求助旁的操作按钮进行分派
</details>

<details>
<summary><b>审核员 — 信息审核</b></summary>

1. 登录 → 进入审核工作台
2. 按紧急度排序的审核队列
3. 超时求助红色高亮 + SLA 倒计时
4. AI 标记的问题求助优先处理
5. 通过 → 进入调度池 / 驳回 → 填写原因
</details>

<details>
<summary><b>救援队 — 执行任务</b></summary>

1. 登录 → 进入任务列表
2. 查看分配给本队的任务
3. 按顺序更新状态：接单 → 出发 → 到达 → 施救 → 完成
4. 如无法完成可拒单或请求增援
</details>

## 技术架构

| 层级 | 选型 | 说明 |
|------|------|------|
| 后端框架 | Go 1.22 + Gin | RESTful API，单体架构，service 层接口化便于拆微服务 |
| 数据库 | PostgreSQL 16 + PostGIS | GIST 空间索引，双坐标策略（精确+偏移） |
| 前端框架 | React 18 + Next.js 14 (App Router) | SSR/SSG，按角色自动路由分流 |
| UI 框架 | Tailwind CSS + daisyUI | 浅天蓝色主题，响应式适配（≤768px 移动端 / >1024px 桌面端） |
| 实时通信 | SSE (MVP) → WebSocket (完整版) | 任务状态变更实时推送，指挥看板自动刷新 |
| 对象存储 | MinIO | S3 兼容 API，可无痛迁移云存储 |
| 缓存 | Redis 7 (MVP 可选) | MVP 阶段 PostgreSQL 承担全部；完整版做热数据加速 |
| 部署 | Docker Compose + Coolify | MVP 单机部署 ($5/月 VPS)，可平滑演进到 K8s |
| 监控 | Prometheus + Grafana (完整版) | MVP 用 `docker stats` 即可 |

### 项目结构

```
├── backend/               # Go + Gin API 服务
│   ├── cmd/server/        # 入口，依赖注入链
│   ├── internal/
│   │   ├── handler/       # HTTP 请求处理 (auth/disaster/help/review/dispatch/task/team/upload/events)
│   │   ├── service/       # 业务逻辑层 (全部接口化，易于 mock 和替换)
│   │   ├── repository/    # 数据访问层 (PostgreSQL 实现，全部接口化)
│   │   ├── middleware/     # JWT/CORS/RBAC/slog日志/限流
│   │   ├── model/         # 数据模型 (10角色RBAC + 11状态任务状态机)
│   │   ├── worker/        # 后台 Worker (AI预审/超时告警/归档)
│   │   └── config/        # 环境变量配置 + 自动迁移/种子
│   ├── migrations/        # SQL (001_init.sql 建表 + 002_seed.sql 种子)
│   └── pkg/push/          # 通用推送接口 (Provider 抽象层)
├── frontend/              # Next.js 前端

│       ├── admin/         # 管理后台 (dashboard/review/tasks/teams/disasters)
│       ├── help/          # H5 求助端 (submit/ID追踪)
│       ├── team/          # 救援队端 (任务列表)
│       └── login/         # 登录页
├── docs/                  # API 文档
├── scripts/               # 运维脚本 (start.sh)
├── requirements/          # 需求 + 方案 + 任务 (开发文档)
├── docker-compose.yml     # PostgreSQL + Redis + MinIO + Backend + Frontend
└── README.md
```
