# 灾害应急调度系统 — 需求文档

> **版本**: v1.1 | **日期**: 2026-07-14 | **状态**: 需求评审通过

---

## 1. 背景与问题定义

### 1.1 为什么需要这个系统

**灾害是持续发生的，不是偶尔的。**

从 2019 年到 2026 年，中国几乎每年都发生多起不同类型的重大灾害：
- 地震（长宁、泸定、积石山、日喀则）、特大洪涝（长江淮河、郑州 7·20、京津冀）、超强台风（利奇马、摩羯）、重大事故（响水爆炸、长沙自建房倒塌、梅大高速塌方）……
- 2024 年全国自然灾害造成 **9413 万人次受灾**，因灾死亡失踪 **856 人**，直接经济损失超 **3000 亿元**。

**每次灾害后，救助环节都面临同样的结构性困境：**

| 痛点 | 表现 |
|------|------|
| **求助信息高度分散** | 灾后大量求救信息散落在微信/微博/QQ/抖音等社交渠道，无法统一收拢、无法有效响应 |
| **协作工具效率低下** | 在线文档作为"救命文档"虽是创举，但全员编辑混乱、移动端体验差、富媒体支持弱，不适合做任务调度 |
| **指挥缺乏全局视角** | 分散的信息难以纳入统筹，指挥方对灾情态势的掌握是模糊、片面的 |
| **每次重头开始** | 每次新灾情，熟悉的组织动员又从头来一遍，难以复用过往经验 |
| **触达覆盖率不足** | 预警和求助入口无法高效触达真正需要帮助的群体 |

### 1.2 核心机会

1. **QQ + 微信覆盖绝大多数中国人**，是独特的高触达基础设施
2. 灾害有其**模式化特征**——地震引发滑坡/危化品泄漏、洪涝引发溃坝/疫情——可构建为预案知识
3. 多次灾害中积累了丰富的实操经验，具备**系统化沉淀**的条件

### 1.3 系统定位

构建一套 **开放、可复用、全链路** 的灾害应急调度系统，连接三个核心要素：

```
需要救助的人 ←→ 救援力量（救援队/志愿者） ←→ 物资资源
```

## 2. 用户角色与核心场景

### 2.1 角色定义

系统涉及 **10 类角色**，按与救援核心流程的关系分为三层：

**第一层：救援核心参与者**

| 角色 | 核心需求 | 使用端 |
|------|---------|--------|
| **受灾群众** | 接收预警 → 快捷求救 → 查看救援进度 → 获取自救知识 | 小程序 / H5 |
| **救援队** | 接收调度指令 → 导航到救援点 → 更新任务状态 → 团队内部协调 | 小程序 / App |
| **志愿者** | 查看附近求助 → 认领力所能及的任务 → 上报现场信息 → 签到打卡 | 小程序 / H5 |
| **指挥员（总指挥/分区指挥）** | 全局态势感知 → 资源调度决策 → 预案管理 → 灾情结束判定 | Web 管理后台/大屏 |

**第二层：运营支撑角色**

| 角色 | 核心需求 | 使用端 |
|------|---------|--------|
| **信息审核员** | 求助真伪验证 → 去重合并 → 恶意信息处理 → 谣言标记 | Web 管理后台 |
| **物资管理员** | 仓库库存管理 → 供需匹配 → 出入库登记 → 物流跟踪 | Web / 小程序 |
| **值班员** | 灾害实例审核开通/关闭 → 预警响应确认 → 告警处理 → 交接班记录 | Web 管理后台 |

**第三层：外围参与角色**

| 角色 | 核心需求 | 使用端 |
|------|---------|--------|
| **捐赠者** | 在线捐款/捐物 → 查看资金物资流向 → 接收反馈 | H5 / 小程序 |
| **系统管理员** | 用户权限管理 → 系统配置 → 字典维护 → 数据归档 | Web 管理后台 |

### 2.2 角色设计说明

**为什么需要"分区指挥"？**
大范围灾害（如 2023 年京津冀洪涝涉及多个省市）不可能由一个指挥员管理所有区域。每个行政区应独立调度本辖区内的救援力量，但总指挥可以跨区协调重大资源调配。

**为什么需要"值班员"？**
灾害应急是 7x24 小时的。预警可能在半夜触发，换班时需要有明确的信息交接机制，确保不会"掉棒"。

**为什么需要"信息审核员"？**
灾害期间信息噪音极高——有重复上报、有误报、甚至有恶意造谣。进入调度池的求助必须是经过验证的，否则会浪费宝贵的救援资源。

**救援队与民间救援力量**
救援队分为注册救援队和民间救援力量两类。民间救援力量（如郑州 7·20 暴雨中的"郑州救援车队"等自发组织）可在系统中登记为救援队并增加"民间救援力量"标记，享有与注册救援队相同的任务接收和调度能力，但在认证标识上有所区分。

## 3. 核心业务流程

### 3.1 全流程概览

一次完整的灾害应急响应分为六个阶段：

```
预警阶段 → 求助上报阶段 → 信息核实阶段 → 救援调度阶段 → 物资保障阶段 → 闭环归档阶段
```

### 3.2 阶段一：灾害预警与通知

**触发条件**：外部预警系统（地震预警网、气象台等）发出预警信号

**业务动作**：
1. 系统接收预警信息（灾害类型、等级、影响区域、预计时间）
2. 根据影响区域划定地理围栏，确定推送目标人群
3. 通过 QQ/微信向围栏内用户推送预警通知（置顶+强提醒）
4. 预警通知中包含求助入口，引导受灾群众进入上报流程

**推送策略**：
- 按灾害等级分级：红色/橙色预警 → 强制置顶通知；黄色/蓝色 → 普通通知
- 受灾区域外的人群不推送（避免信息污染）
- 预警确认解除后，推送解除通知

### 3.3 阶段二：求助信息上报

**入口**：预警通知中嵌入的统一求助入口（小程序/H5）

**核心体验目标**：让求助者在最短的时间内完成有效求助。

**求助流程**：
1. 系统自动识别当前灾害类型，加载对应的求助表单
   - 地震：被困/受伤/倒塌/失联
   - 洪涝：被困/缺水/缺食/需要转移
   - 台风：被困/建筑受损/需要转移
   - 疫情：缺药/需就医/缺生活物资
   - 通用模式：自定义描述
2. 自动采集：GPS 位置（双源定位）、设备信息、用户账号信息
3. 求助者选择：紧急程度（紧急/一般/轻微）、受影响人数、具体需求描述
4. 可选补充：现场照片/视频/录音
5. 提交后显示状态追踪入口，可查看"已提交→已核实→救援中→已获救"的进度

**极端场景设计**：
- **弱网/离线**：求助内容先在本地暂存，网络恢复后自动提交
- **基站大面积损坏**：系统本身保持运行，但告警/推送等无法通过基站触达。救援力量到达后可补充录入信息。有条件时可接入卫星短信/SOS 等补充通信通道（优先级可放低）。
- **紧急求助**：系统预设"一键求救"按钮，单点即发送，自动附带所有可采集信息，适用于被困/受伤等刻不容缓的场景

### 3.4 阶段三：信息核实与去重

**为什么需要独立审核环节**：同一场灾害可能有数十万条求助，其中大量是重复的（同一个人或同一家庭多次上报）、部分可能是虚假的。直接将这些信息送入调度池会导致救援资源严重浪费。

**审核流程**：
1. **AI 预审**：自动标记明显重复（相似度>90%）、异常坐标（海洋/境外）、含敏感词、同设备短时间大量提交等
2. **人工审核**：AI 无法判断的落入人工审核队列
3. **去重合并**：确认是同一事件的多条求助，合并为一条主记录，但保留原始上报计数（用于统计该区域的真实求助密度）
4. **虚假标记**：恶意信息降权或移除，记录提交者特征

**审核 SLA**：
- 紧急求助：提交后 **5 分钟内**完成审核
- 一般求助：提交后 **30 分钟内**完成审核
- 轻微求助：提交后 **2 小时内**完成审核

### 3.5 阶段四：救援调度与执行

**这是系统的核心引擎。**

**调度流程**：
1. 审核通过的求助进入"调度池"（按紧急度、距离、时间排序的地图+列表视图）
2. 系统基于救援力量的位置、能力类型（水域/山地/医疗/破拆）、当前负载生成分派建议
3. 分区指挥确认或调整分派，总指挥可跨区协调重大资源
4. 救援队/志愿者接收任务，确认接单

**任务状态跟踪**：
```
待分配 → 已分配 → 已接单 → 前往中 → 施救中 → 已完成
                                                  → 无法完成
                                                  → 需增援
```

**调度过程中的关键能力**：
- 救援队到达现场后可回传照片/视频/文字更新
- 救援队到达后可确认求助真伪（虚假求助降权）
- 超时未响应/未完成自动告警，升级指挥层级
- 救援队可主动拒单（附原因）或转交给更合适的队伍
- 指挥员可对同区域同类型的求助批量指派

### 3.6 阶段五：物资调度与保障

救助不仅仅是把人从危险中救出，还包括后续的物资保障。

**触发方式**：
- 救援队现场反馈物资需求
- 求助信息中明确提出的物资需求
- 指挥员根据态势判断主动调配

**物资流转链路**：
```
需求汇总 → 供需匹配 → 调拨出库 → 物流运输 → 送达签收
```

**核心管理对象**：
- 仓库（位置、品类、库存、联系人）
- 物资（名称、规格、数量、保质期、类型）
- 调拨单（来源仓库 → 目的地 → 物资明细 → 运输状态）
- 签收记录（谁收到、收到什么、什么时候）

**关键阈值**：
- 关键物资（饮用水/食品/药品/救生衣等）低于安全库存时自动告警
- 保质期物资（食品/药品）临期自动提醒

### 3.7 阶段六：灾情结束与归档

**结束判定**：总指挥确认所有分区进入恢复期后，人工判定灾情结束

**结束后的动作**：
1. 求助入口关闭
2. 未完成的救援任务强制结单或转常态处理
3. 生成灾情总结报告（统计：求助总数/独立求助数/完成率/获救人数/物资消耗/救援力量投入）
4. 数据转入归档状态（在线可查，不可修改）

**归档策略**：灾后数据在线保留 1 个月，期间可人工提前归档；超 1 个月自动归档

## 4. 核心功能需求（按优先级分层）

### 4.1 MVP——最小可用版本（第一阶段）

**目标**：覆盖一次典型灾害的完整求助→调度→完成闭环，证明系统核心价值。

| 功能 | 说明 |
|------|------|
| **灾害实例创建** | 值班员可手动创建灾害实例（灾害类型、影响区域、启动预案） |
| **求助上报入口** | 小程序/H5 的标准化求助表单，自动采集位置+设备信息，支持按灾害类型切换 |
| **求助状态追踪** | 求助者端可查看"已提交→已核实→救援中→已获救"进度 |
| **信息审核** | 审核员工作台：去重、合并、驳回、通过；AI 辅助标记明显异常 |
| **求助池与任务分配** | 地图+列表视图的求助池，指挥员可手工指派给救援队 |
| **任务状态管理** | 救援队端：接收指令、更新状态（接单→前往→到达→施救→完成） |
| **基础指挥看板** | 求助点热力地图 + 救援力量分布 + 核心指标（求助数/已响应/完成率） |
| **救援队注册** | 救援队信息录入、服务区域、能力类型标注；支持注册救援队和民间救援力量两种标记 |

### 4.2 完整版——功能完善（第二阶段）

**目标**：覆盖多角色、多场景，形成完整的应急响应体系。

| 功能 | 说明 |
|------|------|
| **预警推送对接** | 对接外部预警系统，自动触达 + 地理围栏精准推送。设计通用推送接口，支持 QQ/微信/短信/语音电话等多平台按需接入 |
| **智能分派建议** | 基于位置、能力、负载的自动分派建议 |
| **物资管理** | 仓库→入库→出库→调拨→签收全链路 |
| **志愿者模块** | 注册认证、技能标签（预设标签选择 + 自填补充）、查看附近求助、自主认领任务 |
| **募捐模块** | 在线捐款/捐物、资金物资流向透明 |
| **分区指挥** | 按行政区划分区管理，分区指挥只看自己辖区 |
| **指挥大屏** | 适配大屏的全面监测面板，多图层（卫星/气象/交通）叠加 |
| **突发/弱网求助** | 离线暂存+自动补传 |
| **预案管理** | 灾害类型→预案模板，一键激活 |
| **科普知识库** | 灾害类型分类知识、场景化自救指南 |

### 4.3 理想版——增强能力（第三阶段）

| 功能 | 说明 |
|------|------|
| **多灾情并行管理** | 同时管理多场灾害，数据互不干扰 |
| **历史趋势对比** | 过往数据归档，支持横向对比分析 |
| **灾情报告自动生成** | 日报/总结报告，导出 PDF/Word |

## 5. 关键业务决策

### 5.1 触达策略

**核心假设**：QQ + 微信能覆盖绝大多数需要帮助的人。

**决策结论**：
- **多平台通用推送接口**：设计通用的推送接口层，支持 QQ、微信、短信、语音电话等多平台按需接入。平台特殊能力（如强提醒、置顶）需要平台侧配合开发，但系统本身不硬编码依赖任何单一平台。具体平台的接入可与相关方后续洽谈。
- **弱势群体补充触达**：对于不使用智能手机的群体（老人、部分偏远地区人群），系统需支持短信/语音电话作为补充触达通道。

### 5.2 救援力量认证

**决策结论**：
- **救援队准入**：救援队注册需经过基本认证审核，防止冒名救援队进入系统。救援队可选择标记为"注册救援队"或"民间救援力量"。
- **志愿者技能标签**：提供预设标签供选择（如"水域救援""医疗急救"），同时支持自定义补充填写，无需强制验证。

### 5.3 求助真伪判断

**决策结论**：
- **信用分体系**：设计信用分体系替代简单黑名单。当求助被标记为虚假时降低提交者信用分，并给出明显标记。追踪粒度为设备级+账号级+手机号级（如可获取）。
- **孤立上报处理**：对于偏远地区单人被困等无人佐证的孤立上报，不可忽略，应在系统中标记为"孤立上报"标签，供审核员和指挥员参考。

### 5.4 数据隐私

- 受灾群众的位置信息如何处理？是否需要在公开地图上做偏移？**（待决策）**
- 求助者的个人身份信息（手机号、账号）在调度过程中哪些角色可见？**（待决策）**

### 5.5 系统边界

**决策结论**：
- **系统定位**：系统定位为"尽力而为的工具"，但在架构上需考虑灾害规模弹性，确保在大规模灾害（如全国性疫情、数百万求助）时系统核心功能不崩溃，能保底运行。
- **通信中断场景**：通信完全中断时系统保持运行状态，但告警/推送等无法通过基站触达；救援力量到达后可补充录入信息。卫星通信等补充通道可作为远期能力，优先级可放低。

## 6. 成功指标（非功能性）

> 排除技术指标（QPS/响应时间等），只描述业务结果。

### 6.1 效率提升
- 求助者从打开入口到完成求助提交：**平均不超过 60 秒**（MVP 阶段可放宽）
- 从求助提交到审核通过进入调度池：紧急求助不超过 **5 分钟**，一般求助不超过 **30 分钟**
- 从求助进入调度池到任务分配给具体救援队：不超过 **15 分钟**

### 6.2 触达率
- 灾害预警推送在目标区域的触达率：**>80%**（基于 QQ/微信的覆盖假设）
- 求助入口的用户知晓率：推送后的打开率 **>30%**

### 6.3 数据质量
- 求助去重率（合并为同一事件的占比）：系统应识别出 **>90%** 的重复求助
- 虚假求助占比：经审核过滤后进入调度池的虚假求助 **<5%**

### 6.4 透明度
